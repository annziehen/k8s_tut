# Приложение для Kubernetes (Учебный проект)

Данный репозиторий представляет собой учебное приложение для изучения Kubernetes (K8S). 

Приложение состоит из двух микросервисов:
- **Frontend** - Angular-приложение. В рамках K8s оно должно быть сбилжено в статику и раздаваться веб-сервером (например, базовым образом Nginx).
- **Backend** - Go-приложение, предоставляющее API. При GET-запросе на `/api/v1/status` возвращает JSON `{"status": "running"}`.

---

## Замечание по архитектуре и Kubernetes особенностям


1. Фронтенд делает API-вызовы по относительному пути (`/api/v1/status`). Это позволяет обойти настройки междоменных запросов (CORS) при запуске в Kubernetes, так как браузер будет думать, что фронтенд и бекенд находятся по одному адресу. Описание того, как это работает внутри, приведено ниже.

2. Вам понадобится какой-либо http-сервер (чаще всего берется легковесный `nginx:alpine` или `httpd`) в поде фронтенда *только для того, чтобы раздавать статические собранные HTML/JS файлы*. Однако **описывать сложную логику reverse proxy встроенными конфигами самого Nginx не нужно**. 

Всю маршрутизацию берет на себя ресурс **Ingress** в Kubernetes:
1. Вы устанавливаете Ingress Controller (например, nginx-ingress) на кластер.
2. В файле манифеста Ingress вы описываете правила:
   - Если путь начинается с `/api/v1/` — направлять трафик на внутренний Service бекенда (`ClusterIP`).
   - Если путь другой (например, `/`) — направлять трафик на Service фронтенда.

При такой архитектуре:
- Backend остается полностью скрыт извне (`ClusterIP`), и доступен только внутри кластера.
- Frontend и Backend доступны "извне" через Ingress, который сам решает, куда отправить запрос.

## Задача

Главная цель проекта — поднять кластер Kubernetes так, чтобы удовлетворялись следующие условия:
1. **Backend** должен быть запущен в отдельном поде.
2. **Backend** **не должен быть доступен извне** кластера напрямую (доступен через сервис типа `ClusterIP`).
3. **Frontend** должен быть сбилжен в статические файлы и сервиться веб-сервером в поде-фронтенде.
4. **Связь Frontend и Backend**: на уровне Kubernetes настроить Ingress (или другие механизмы маршрутизации), чтобы фронтенд, посылая запросы на маршрут API, мог успешно получить ответ бекенда и отобразить сообщение **"Server is accessible"**.

## Инструкции по запуску (Локально)

Для разработки и проверки работы приложений вне Kubernetes:

### Backend (Go)
1. Убедитесь, что у вас установлен **Go** (рекомендуется 1.20+).
2. Перейдите в папку бекенда и запустите сервер:
```bash
cd backend
go run server.go
```
Сервер будет доступен на `http://localhost:8080`.
Проверка: `curl http://localhost:8080/api/v1/status`

### Frontend (Angular)
1. Убедитесь, что у вас установлены **Node.js** и **npm**.
2. В отдельном терминале перейдите в папку фронтенда, установите зависимости и запустите локальный сервер разработки:
```bash
cd frontend
npm install
npm start
```

*Примечание:* Так как фронтенд запрашивает `/api/v1/status` по относительному пути (а его dev-сервер запускается на `localhost:4200`), запросы пойдут на `localhost:4200/api/v1/status` и упадут с ошибкой 404. Для локальной разработки Angular без K8s нужно использовать [proxy.conf.json](https://angular.io/guide/build#proxying-to-a-backend-server), чтобы dev-сервер Angular сам проксировал эти запросы к бекенду на порт 8080.

## Деплой в Kubernetes (Шаги установки)

Для развертывания приложения в K8s вам потребуется выполнить следующее:
1. **Сборка Docker образов**:
   - `Dockerfile` для Backend (компиляция Go-приложения в бинарный файл).
   - `Dockerfile` для Frontend (сборка `npm run build` и копирование статики в nginx).
2. **Написание манифестов Kubernetes** (`.yaml`):
   - `Deployment` и `Service` (`ClusterIP`) для Backend.
   - `Deployment` и `Service` (`ClusterIP` или `NodePort`) для Frontend.
   - `Ingress` с правилами маршрутизации для `/api` и `/`.
3. **Запуск**:
   ```bash
   kubectl apply -f manifest.yaml
   ```
